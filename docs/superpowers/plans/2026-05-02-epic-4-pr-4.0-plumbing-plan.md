## 3. Task: `metricFields.ts` literal-tuple typing (Zod v4 `z.enum` requirement)

**Files.** Test: append to [packages/schema/src/metricFields.test.ts](packages/schema/src/metricFields.test.ts). Modify: [packages/schema/src/metricFields.ts](packages/schema/src/metricFields.ts).

**Why.** `METRIC_FIELDS` is currently typed `readonly MetricPath[]` via the explicit annotation. Zod v4's `z.enum(values)` requires a **literal tuple** (`readonly [string, ...string[]]`) to derive the key union; passing a widened `readonly MetricPath[]` makes the resulting Zod schema's key type widen to `string`, which **silently defeats the cross-key invariant** that Tier 1 PR 4.1a's `WorkspaceFile.refine()` is supposed to enforce ([spec ¬ß10.2 line 956-958](docs/superpowers/specs/2026-05-01-epic-4-admin-app-design.md)).

The fix is non-breaking: drop the explicit `: readonly MetricPath[]` annotation; let TypeScript infer the literal tuple from the `as const`-ed array; re-derive `MetricPath` from `(typeof METRIC_FIELDS)[number]`. The type union stays identical; consumers expecting `readonly MetricPath[]` still see assignment compatibility (literal tuples are assignable to widened arrays).

**Note.** This bug also affects [spec ¬ß10.2 line 957's WorkspaceFile code block](docs/superpowers/specs/2026-05-01-epic-4-admin-app-design.md) ‚Äî the spec author and the Tier 1 plan author both copied the same broken pattern. The spec is not amended in PR 4.0 (specs are append-only history); the **fix in `metricFields.ts` makes the spec-as-written work correctly** because `z.enum(METRIC_FIELDS)` now narrows. A spec-level note may follow as a separate doc fold.

- [ ] **Step 1: Append failing test** to `packages/schema/src/metricFields.test.ts`:
  ```ts
  // Existing tests cover the array contents; this case pins the LITERAL-TUPLE typing
  // so z.enum(METRIC_FIELDS) narrows to the union of the 12 string literals (and
  // the WorkspaceFile cross-key invariant works as intended).
  import { describe, expect, expectTypeOf, it } from 'vitest'
  import { z } from 'zod'
  import { METRIC_FIELDS, type MetricPath } from './metricFields'

  describe('METRIC_FIELDS literal-tuple typing (PR 4.0)', () => {
    it('z.enum(METRIC_FIELDS) narrows to the MetricPath union', () => {
      const schema = z.enum(METRIC_FIELDS)
      // Runtime: parses every literal, rejects unknown.
      expect(schema.safeParse('snow_depth_cm').success).toBe(true)
      expect(schema.safeParse('not-a-metric').success).toBe(false)
    })

    it('TS: z.enum(METRIC_FIELDS) output type is the MetricPath union, not string', () => {
      const schema = z.enum(METRIC_FIELDS)
      type Inferred = z.infer<typeof schema>
      expectTypeOf<Inferred>().toEqualTypeOf<MetricPath>()
    })

    it('TS: METRIC_FIELDS exports as a readonly tuple, assignable to readonly MetricPath[]', () => {
      // Backwards-compat: existing consumers using `readonly MetricPath[]` annotations still work.
      const _widened: readonly MetricPath[] = METRIC_FIELDS
      expect(_widened.length).toBe(12)
    })
  })
  ```

## 8. Rollback policy

Per [parent spec ¬ß10.4](docs/superpowers/specs/2026-04-22-product-pivot-design.md) + [ADR-0009](docs/adr/0009-dco-exemption-for-dependabot.md): rollback is `git revert <merge-sha>` directly on `main`.
