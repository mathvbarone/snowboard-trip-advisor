## Spec deviations from Epic 3 design (load-bearing — read before Epic 4)

These are deviations the actual implementation made vs. the Epic 3 design spec. Future agents reading the spec need to know which were retconned.

1. **`check-preload-hrefs` scope expanded.** Spec §10.7 originally said "parses `dist/index.html`'s `<link rel="preload">` tags". The implementation also walks `dist/assets/*.js` for runtime-injected `/assets/*.woff2?` URL string literals (because `injectFontPreloads` resolves `?url` imports to JS string literals, not HTML preload tags). The script also rejects hrefs whose resolved path escapes `dist/` (containment guard). Spec §10.7 is updated to reflect this; future agents reading the original deliverable list in PR breakdown §7.12 should consult §10.7 for the actual contract.

2. **PR 3.6 was split into 3.6a/3.6b/3.6c.** The original spec §7.12 packaged these as one "PR 3.6". Execution split per CLAUDE.md atomic-PR discipline. PRs #53 (3.6a), #55 (3.6b), #57 (3.6c) — see §7.12 Status pointer.

3. **PR #56 phantom-merge.** PR #56 (the original 3.6c) was marked `MERGED` by GitHub after its base branch was auto-deleted, but the actual diff did NOT land on `main`. PR #57 re-applied the content. See spec §10.8 for the failure mode and future-agent guidance on detecting it.

4. **`useScrollReset` / `useDroppedSlugs` shipped as stubs in 3.1c.** Spec §7.7 (PR 3.1c) listed these as deliverables, but the actual implementation shipped null-stubs in 3.1c and finalized in 3.6a. Spec §7.7 Status note now mentions this.

5. **`<NoResorts>` defence-in-depth.** PR 3.6a's review surfaced that `<NoResorts>` was orphaned (`cardsSelectors.filterViews` silently no-op'd on unknown country codes). Folded: `filterViews` is now strict (returns `[]` on unmatched country), `<NoResorts>` is wired into the cards view. Plan §1110 covers this.

6. **Bundle budget is warn-mode, gzip total ~195 KB on the seed bundle** — over the 100 KB advisory budget per spec §6.7. Expected pre-tightening; Epic 6 follow-up flips to error-mode and tightens. Don't treat the WARN as a regression.

## Epic 4 — admin app outline (next milestone)

**Goal.** Ship `apps/admin`: a loopback-only Vite SPA + in-process API middleware that lets a single analyst edit `Resort` and `ResortLiveSignal` documents, see live signal field-state (Live / Stale / Failed / Manual), trigger adapter test/sync, and publish the dataset all-or-nothing through the same pipeline as `research/cli.ts publish`.

**Non-goals (Phase 1):**

- No auth (loopback boundary IS the control). `127.0.0.1:5174` only, `strictPort: true`, never built into a production container image.
- No per-resort publish workflow (deferred to Phase 2 when resort count crosses ~25).
- Read-only below the `md` breakpoint (edit controls removed from tab order, not merely disabled).
- No real adapters yet — Epic 4's `Test` / `Sync` actions invoke the 5 stub adapters in `packages/integrations/`. Real adapter wiring lands in Epic 5.

**Architectural primitives (already on main from Epic 3):**

- `packages/schema` document types + `validatePublishedDataset` + `publishDataset` + `loadResortDatasetFromObject`.
- `packages/design-system` for the shared component vocabulary; needs additions: Sidebar, StatusPill (`Live | Stale | Failed | Manual`), Tabs, Toast, Popover, DropdownMenu, TextArea (admin-only per Epic 3 §5.1).
- `packages/integrations` adapter contract + stubs for `Test` / `Sync`.

**New surface (Epic 4 introduces):**

- `apps/admin/src/` SPA (mirroring `apps/public/src/` topology — `lib/`, `state/`, `views/`, `App.tsx`).
- `apps/admin/server/` handler modules (one per `/api/*` route from parent spec §8).
- `apps/admin/vite-plugin-admin-api.ts` middleware that dispatches `fetch('/api/...')` to the server handlers in-process. Same Node process as the dev server (parent spec §3.11).
- `packages/schema/api/` Zod request/response schemas (already mentioned in CLAUDE.md Subagent Review Discipline trigger list — once they exist).
- A new ADR for any non-obvious admin decision (auth model, draft-vs-published preview, edit history surfacing). Default to "no ADR required" until a load-bearing decision arises.

**Likely PR breakdown (to be validated with brainstorm before drafting Epic 4 spec):**

1. **PR 4.1a** — packages/schema/api/ Zod schemas (read-only types first; mutations land later).
2. **PR 4.1b** — Vite admin-api middleware skeleton + `apps/admin/vite.config.ts` + `apps/admin/index.html` + 127.0.0.1 binding.
3. **PR 4.1c** — design-system additions: Sidebar + StatusPill + Tabs + Toast + Popover + DropdownMenu + TextArea.
4. **PR 4.2** — Dashboard + Resorts table.
5. **PR 4.3** — Resort editor (the core): durable + live panels, ModeToggle, FieldValueRenderer, StatusPill wiring.
6. **PR 4.4** — Test / Sync actions (against stubs).
7. **PR 4.5** — Publish workflow (all-or-nothing pipeline reuse).
