## What NOT to do (lessons learned in this session)

1. **Don't modify `App.tsx`'s lazy-import lines.** They're part of the §5.5 frozen interface. PR 3.4 and PR 3.5 fill the body of `views/{matrix, detail}.tsx`; the import paths stay byte-identical.
2. **Don't re-introduce the `useDataset` auto-clear-in-catch pattern** without verifying React 19 has fixed the suspended-by-uncached-promise loop. Read `useDataset.ts` leading comment first.
3. **Don't reuse the `EMPTY_DATASET_ZOD_MESSAGE` literal for other Zod rules** — it's reserved for `validatePublishedDataset`'s `dataset_empty` discrimination. The constant lives in `packages/schema/src/published.ts`.
4. **Don't `cd` chained without absolute paths.** Each Bash tool call has fresh shell state — `cd` doesn't persist. Either chain with `&&` or use `git -C <dir>` / `npm --workspace=<name>`.
5. **Don't trust the dev server on port 5173 was started by your session** — there may be leftover servers from previous sessions / user activity. Always `pkill -f vite` + check `lsof -i :5173` before starting.
6. **Don't `npm run --workspace=@snowboard-trip-advisor/tests-integration ...`.** That workspace isn't in npm's `workspaces` array (only `apps/*` and `packages/*` are). Use `npx vitest run tests/integration/<file>.test.ts` from the worktree root instead.
7. **Don't modify the matrix/detail stub contracts.** They throw deliberately for PR 3.1c so coverage tests assert the throw line. PR 3.4/3.5 replace just the bodies.
8. **Don't put inline `/* v8 ignore */` markers anywhere.** CLAUDE.md "Coverage Rules" forbids them by intent. Extract dead code to its own file and exclude via glob in `vite.config.ts`.
